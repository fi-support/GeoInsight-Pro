This app uses OAuth 2.0 Authorization Code + PKCE via Clearly.Hub. No passwords are ever handled by the app directly — the identity provider manages all credentials.

1
Generate PKCE Challenge
A random code_verifier and its SHA-256 hash (code_challenge) are created client-side before any redirect.
const verifier = generateRandomString(128);
const challenge = await sha256(verifier);
sessionStorage.setItem('pkce_verifier', verifier);
2
Redirect to Identity Provider
The user is sent to the Clearly.Hub Cognito endpoint. Your app never sees the password.
window.location.href =
  `https://auth.clearly.app/oauth2/authorize
  ?response_type=code
  &client_id=${CLIENT_ID}
  &code_challenge=${challenge}
  &code_challenge_method=S256`;
3
Exchange Code for Tokens
After login, Cognito redirects back with a short-lived code. The app exchanges it for an access_token.
const res = await fetch(TOKEN_ENDPOINT, {
  method: 'POST',
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code, code_verifier: verifier,
    client_id: CLIENT_ID
  })
});
const { access_token } = await res.json();
4
Authorized API Access
Every subsequent API call includes the token as a Bearer header. The platform verifies it on every request.
fetch('/api/graphql', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${access_token}`
  },
  body: JSON.stringify({ query, variables })
});

The Billing Component is a hosted UI on Clearly.Hub. Your app redirects the user there; the platform handles subscription selection and payment, then redirects back with a result.

1
Redirect to Billing Component
Construct a base64-encoded payload and append it to the component URL. The origin is where Clearly.Hub will redirect the user back after checkout.
const payload = btoa(JSON.stringify({
  actions: ["SELECT_SUBSCRIPTION"],
  redirect_url: REDIRECT_URI,
  client_id: CLIENT_ID
}));
window.location.href =
  `https://hub.clearly.app/components/${payload}`;
2
User Selects a Plan
Clearly.Hub displays the available subscription plans for your app. The user picks one and confirms. Your app is not involved in this step — no payment code required.
3
Handle the Redirect Back
Clearly.Hub redirects to your origin with a ?data= query parameter containing a base64-encoded subscription result.
const params = new URLSearchParams(location.search);
const raw = params.get('data');
if (raw) {
  const result = JSON.parse(atob(raw));
  // result.subscription.name  → "PROFESSIONAL"
  // result.subscription.externalReference
  // result.subscribedHub._id
  sessionStorage.setItem(
    'subscriptionName',
    result.subscription.name.toUpperCase()
  );
}
Tip: Use externalReference on the subscription model to map Clearly.Hub plan IDs to your own internal tier names (e.g., "starter", "pro", "enterprise").

Actual GraphQL queries and REST calls made by this application. All require Authorization: Bearer <token>.

GQL
DatasetsForHubFilter All datasets visible to the active hub — builds the Owner Hub picker 
datasets(activeHubId: $id, limit: 500) {
  results {
    _id
    ownerHub { _id name }
    resources { format }
  }
}
GQL
DatasetsByOwner WMS / WFS datasets for a selected hub — populates the Dataset picker 
datasets(
  activeHubId: $id
  query: { ownerHubId: $hubId }
) {
  results { _id title resources { format } }
}
GQL
Dataset (by id) Resource URLs & spatial extent — called when adding a layer to the map 
dataset(_id: $id, activeHubId: $id) {
  _id title
  spatial { coordinates }
  resources { url format }
}
GET
WMS GetCapabilities Auto-discovers the layer name before adding WMS to the map 
{url}?service=WMS&request=GetCapabilities
GET
WFS GetFeature Fetches GeoJSON features from a WFS endpoint 
{url}?service=WFS&version=2.0.0
  &request=GetFeature
  &outputFormat=application/json
GET
/api/apps?ssoClientId=… Resolves the Clearly.Hub App ID from the Cognito SSO client

/api/me/app-subscriptions/{appId} Loads the user’s active subscription tier on every login
